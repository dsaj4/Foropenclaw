# 报告分析写作迭代总指示

## 一、流程定位

本流程是一套**可积累、可迭代**的公司研究报告分析写作系统，通过"分析-提取-复用-沉淀"的闭环机制，实现报告写作能力的持续进化。

---

## 三、四阶段核心任务

| 阶段 | 核心任务 | 输入 | 输出 | 关键产出物 |
|------|---------|------|------|-----------|
| **阶段一：框架分析** | 解构优质报告，提取可复用要素 | 标杆研究报告 | 结构化分析报告 | 框架要素清单、特殊框架库、风格参数 |
| **阶段二：框架重构写作** | 基于提取要素撰写新报告 | 新公司数据 + 阶段一产出 | 新报告初稿 | 三级目录、总起句、论证段、风格校准稿 |
| **阶段三：一致性校验** | 验证新报告与提取标准的符合度 | 新报告初稿 + 阶段一标准 | 校验报告 | 一致性评分、修订建议 |
| **阶段四：知识沉淀** | 将新经验转化为可复用资产 | 本次迭代全部产出 | 更新的框架资产 | 新版本框架库、新增特殊框架、优化参数 |

## 四、迭代循环机制

### 4.1 单次迭代流程（针对单一新报告）

```
标杆报告 ──→ 框架分析 ──→ 框架重构 ──→ 一致性校验 ──→ 定稿报告
                │                              │
                └────────→ 知识沉淀 ←─────────┘
```

### 4.2 持续积累流程（跨报告迭代）

```
第N次迭代                    第N+1次迭代
    │                            │
    ▼                            ▼
┌─────────┐                ┌─────────┐
│报告A分析 │────┐    ┌────→│报告B分析 │
│(产出v1) │    │    │     │(使用v1) │
└─────────┘    │    │     └─────────┘
               │    │          │
               ▼    │          ▼
          ┌─────────┐│     ┌─────────┐
          │框架库v1 │┘     │框架库v2 │
          │(基础版) │─────→│(增强版) │
          └─────────┘      └─────────┘
                               │
                               ▼
                          ┌─────────┐
                          │框架库vN │
                          │(成熟版) │
                          └─────────┘
```

## 五、核心资产定义

### 5.1 报告结构框架（标准化模板）

**定义**：经过多轮迭代沉淀形成的、可跨报告复用的标准化章节体系

**组成**：
- **固定框架**：所有公司报告必须包含的基础章节（公司概况、财务分析、风险提示、投资要点）
- **特殊框架库**：针对特定行业/公司类型可选用的扩展章节集合
- **框架关联规则**：章节间的逻辑依赖与排列顺序规范

**版本管理**：
- 主版本号：框架结构重大调整（如新增一级章节）
- 次版本号：特殊框架库扩展（如新增行业特定章节）
- 修订号：参数微调（如句长标准调整）

### 5.2 框架要素清单（单次分析产出）

**定义**：针对特定标杆报告解构出的可复用要素集合

**组成**：
- 目录结构树（含层级关系）
- 特殊框架清单（含适用条件）
- 总起句功能类型库
- 论证逻辑模式库
- 语言风格参数集

### 5.3 知识积累规则

**什么应该被沉淀**：
1. **通用性强**的要素（如"财务分析"的标准维度）
2. **验证有效**的特殊框架（在多份报告中成功复用）
3. **量化稳定**的风格参数（经多份报告验证的指标）

**什么不应该被沉淀**：
1. 特定公司的**专有数据**
2. 单次报告的**临时性表达**
3. 未经**二次验证**的特殊框架

## 六、执行指令集

### 6.1 启动条件判断

| 场景 | 执行路径 |
|------|---------|
| 首次执行，无历史框架 | 执行完整四阶段流程，初始化框架库v1.0 |
| 已有框架库，分析新标杆报告 | 执行阶段一，对比现有框架，更新框架库 |
| 已有框架库，撰写新报告 | 直接使用阶段二，跳过阶段一 |
| 框架库版本过期 | 执行阶段一重新校准，升级框架库版本 |

### 6.2 质量门禁

每个阶段设置通过标准，未达标需返回修订：

| 阶段 | 通过标准 |
|------|---------|
| 阶段一 | 框架要素提取完整度≥90%，特殊框架标注适用条件 |
| 阶段二 | 固定框架100%继承，特殊框架适配合理 |
| 阶段三 | 一致性评分≥85分，关键维度无红色项 |
| 阶段四 | 沉淀资产经审核，版本更新记录完整 |

## 八、使用示例

### 示例1：首次建立框架库

```
输入：一份优质标杆报告（如"腾讯控股研究报告"）
执行：阶段一 → 阶段二 → 阶段三 → 阶段四
产出：
  - 框架要素清单_腾讯控股_20260304.md

```

执行：阶段二 → 阶段三
产出：
  - 新报告初稿_[目标公司]_20260304.md
  - 一致性校验报告_[目标公司]_20260304.md

### 示例3：框架库迭代升级

```
输入：新标杆报告（如"阿里巴巴研究报告"）+ 报告结构框架_v1.0
执行：阶段一（对比分析）→ 阶段四（框架更新）
产出：
  - 框架要素清单_阿里巴巴_20260304.md
  - 报告结构框架_v1.1_20260304.md（新增电商特殊框架）
```

## 九、关键原则

1. **先分析后写作**：未经框架分析，不得直接撰写报告
2. **先继承后创新**：优先复用已验证框架，再考虑新增
3. **量化优先**：风格参数必须可测量、可验证
4. **持续沉淀**：每次迭代必须产出可复用资产
5. **版本管理**：框架库变更必须记录版本和变更说明
